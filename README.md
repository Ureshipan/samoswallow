# samoswallow

Self-hosted мини-PaaS: «самосвал» заглатывает гит-репозитории и выкатывает из них
контейнеризированные сервисы с доступом по поддомену, мониторингом и управлением
через веб-интерфейс.

Полное описание архитектуры и дорожная карта — в [`CONCEPT.md`](./CONCEPT.md).

## Стек

Rust (axum + tokio + sqlx/SQLite) · Docker (через `bollard`) · Caddy (wildcard TLS).

## Разработка

Удобнее всего через dev-скрипты (фоновый запуск на `:8088`, свой state в `./.dev`):

```sh
./scripts/dev-start.sh     # собрать и поднять в фоне
./scripts/dev-stop.sh      # остановить
# другой порт:  SWALLOW_LISTEN=127.0.0.1:9000 ./scripts/dev-start.sh
```

Или вручную:

```sh
cargo build
SWALLOW_LISTEN=127.0.0.1:8088 cargo run -p swallowd
curl http://127.0.0.1:8088/healthz
```

Конфигурация через переменные окружения (`SWALLOW_*`), см. `crates/swallowd/src/config.rs`.

## Развёртывание на сервере

```sh
cargo build --release                # бинарь -> target/release/swallowd
sudo ./scripts/install.sh            # ставит бинарь, systemd-юниты, Caddy, конфиг
```

`install.sh` делает: кладёт бинарь в `/usr/local/bin/swallowd`, ставит и
запускает Docker (если нет) и Caddy, создаёт конфиг `/etc/samoswallow/swallowd.env`
и каталог состояния `/var/lib/samoswallow`, регистрирует и запускает systemd-юниты
`swallowd` и `swallow-caddy`.

После установки интерфейс доступен по адресу из `SWALLOW_LISTEN` (по умолчанию
`http://127.0.0.1:8080/`). Логин — пользователь `admin`.

### Где задавать пароль, порт и прочие переменные

Все настройки — это переменные окружения в файле **`/etc/samoswallow/swallowd.env`**
(его создаёт `install.sh`). После правки файла перезапусти сервис:

```sh
sudo nano /etc/samoswallow/swallowd.env
sudo systemctl restart swallowd
```

| Переменная               | Назначение                                        | По умолчанию                         |
|--------------------------|---------------------------------------------------|--------------------------------------|
| `SWALLOW_LISTEN`         | адрес и **порт** веб-интерфейса/API               | `127.0.0.1:8080`                     |
| `SWALLOW_ADMIN_PASSWORD` | **пароль** пользователя `admin`                   | если пусто — генерируется в лог      |
| `SWALLOW_BASE_DOMAIN`    | базовый домен; `app` отдаётся на `app.<домен>`    | `localhost`                          |
| `SWALLOW_DATABASE_URL`   | путь к SQLite-базе                                 | `sqlite:///var/lib/samoswallow/state.db` |
| `SWALLOW_STATE_DIR`      | каталог рабочих данных (клоны репо, сборки)        | `/var/lib/samoswallow`               |
| `SWALLOW_CADDY_ADMIN`    | URL Admin API Caddy                               | `http://127.0.0.1:2019`              |
| `SWALLOW_LOG`            | уровень логов (`info`, `debug`, …)                | `info`                               |

> Если `SWALLOW_LISTEN` смотрит наружу (`0.0.0.0:…`), обязательно задай
> `SWALLOW_ADMIN_PASSWORD` — иначе морда будет открыта в сеть с авто-паролем
> только из логов. Рекомендуется держать `SWALLOW_LISTEN=127.0.0.1:…` и
> публиковать интерфейс через Caddy.

### Доступ к самому самосвалу по поддомену

Сам интерфейс самосвала доступен без указания порта — по поддомену третьего
уровня **`samoswallow.<SWALLOW_BASE_DOMAIN>`** (например `samoswallow.example.com`).
swallowd при старте регистрирует этот маршрут в Caddy, проксируя на адрес из
`SWALLOW_LISTEN` (порт всегда берётся из конфига — там, где демон реально слушает).
Поэтому можно держать `SWALLOW_LISTEN=127.0.0.1:8080`, не публиковать порт наружу,
и заходить в морду по поддомену. Нужен лишь DNS-запись/wildcard на сервер.

### Управление сервисом

samoswallow и его reverse-proxy — два systemd-юнита: `swallowd` и `swallow-caddy`.

```sh
sudo systemctl status swallowd          # состояние демона
sudo systemctl restart swallowd         # перезапуск (после правки конфига)
sudo systemctl stop swallowd            # остановить
journalctl -u swallowd -e -f            # логи демона (живой хвост)

sudo systemctl status swallow-caddy     # состояние Caddy
journalctl -u swallow-caddy -e          # логи Caddy
```

Статус Caddy также виден прямо в интерфейсе — бейдж «Caddy: онлайн/офлайн»
в шапке. Проверить вручную: `curl -s http://127.0.0.1:2019/config/` (ответ =
Caddy жив).

### Обновление до новой версии

Типичный цикл: запушил изменения с рабочей машины → на сервере подтянул и
применил. На **сервере**:

```sh
cd /path/to/samoswallow
git pull                         # подтянуть новый код
cargo build --release            # пересобрать бинарь
sudo ./scripts/install.sh        # переложить бинарь + юниты и перезапустить сервисы
```

Это всё, что нужно для обновления **самого самосвала**. `install.sh`
идемпотентен: `swallowd.env` и база в `/var/lib/samoswallow` не трогаются,
обновляются только бинарь и systemd-юниты, после чего `swallowd` и
`swallow-caddy` перезапускаются. При изменении схемы БД миграции применяются
автоматически при старте.

Что происходит при перезапуске:

- **`swallowd`** при старте сам поднимает базовый сервер Caddy (`:80`/`:443`) и
  сверяет его настройки — то есть маршрутизация готова сразу, без первого деплоя.
- **Маршруты приложений переживают перезапуск**: Caddy запускается с
  `--resume` и восстанавливает последний применённый конфиг, swallowd его не
  затирает.
- **Приложения переживают перезагрузку сервера.** Контейнеры запускаются с
  политикой `unless-stopped`, поэтому Docker сам поднимает их после ребута. В
  дополнение swallowd при старте сверяет состояние: поднимает остановленные
  контейнеры, помечает в БД исчезнувшие инстансы как `stopped` (чтобы UI не
  показывал «active» для мёртвых), и заново применяет маршруты Caddy. Это лечит
  и приложения, задеплоенные старой версией (со старой политикой рестарта).

Если правил только бинарь и не хочешь гонять весь скрипт — достаточно заменить
бинарь и перезапустить демон (Caddy перезапускать не обязательно):

```sh
sudo install -m755 target/release/swallowd /usr/local/bin/swallowd
sudo systemctl restart swallowd
```

> **Важно:** обновление самосвала **не** пересобирает уже задеплоенные
> приложения. Если изменился код самого приложения (или его `swallow.yaml`),
> задеплой его заново — кнопкой **Deploy** в UI или через API:
> `curl -u admin:ПАРОЛЬ -X POST http://127.0.0.1:8080/api/apps/<id>/deploy`
> (`<id>` — числовой идентификатор приложения, см. `GET /api/apps`).

### Удаление

```sh
sudo ./scripts/uninstall.sh           # снести бинарь, юниты, конфиг (данные сохранить)
sudo ./scripts/uninstall.sh --purge   # + удалить состояние /var/lib/samoswallow
```

## Статус

**Фаза 1 (MVP) — готова.** **Фаза 2 — в работе.** Работает:

- HTTP JSON API + серверный веб-интерфейс (maud SSR), аутентификация (argon2).
- Деплой из git: clone → сборка образа (Docker/bollard) → запуск инстанса →
  роут поддомена через Caddy → ротация старых инстансов.
- Автодеплой по webhook'ам (push в ветку, проверка HMAC-подписи).
- Откаты на прошлый билд без пересборки.
- Управление инстансами: restart / stop, прямые ссылки, логи в UI.
- Постоянное хранилище: папка с хоста монтируется в контейнер (по умолчанию в
  `/data`) — для баз данных и файлов, переживающих пересборки и перезагрузки и
  доступных с хоста напрямую. Задаётся при создании приложения и в его настройках.
- Восстановление после перезагрузки сервера (см. «Что происходит при перезапуске»).
- Метрики во времени: графики CPU/RAM (фоновый сэмплер + retention).
- Бейдж статуса Caddy в интерфейсе.
- SQLite-состояние с миграциями; установка/удаление/обновление скриптами.

Осталось по Фазе 2: несколько инстансов на приложение + балансировка. См.
`CONCEPT.md`.
